# WL-0043 -- DOS-3.3 parity Wave 4: p96i (ANSI.SYS wired into CON) + the FAT-cache share (option B), with an ADR-by-committee on the kernel window

**Type:** Feature push + a committee-ratified memory decision that the orchestrator's grading overturned in implementation. **Date:** 2026-06-20. **Branch:** command-com-default. Continues WL-0042 (Wave 3).

## Context

p96i is the last doable `40oq` capstone dep (x3mh's CON wiring). The ANSI FSM (ansi.c, x3mh Wave 1) + 6zd9's `g_ansi_enabled` flag + the single CON output path teed it up. Wiring it grew the shell kernel past the 128 KiB window -- the recurring kernel-window pressure reached its limit and forced a serious memory decision.

## What changed

- **`initech-p96i` -- ANSI.SYS wired into the CON output path.** The console (console.c/h) gained a current attribute (`cur_fg`/`cur_bg`, default == today's ink/paper so escape-free output is pixel-identical) + `console_set_attr`/`set_cursor`/`get_cursor`/`erase_display`/`erase_line` + the IBM 16-colour CGA palette. int21.c's `con_putc` (the ONE CON choke point) now feeds bytes to `ansi_feed` when ANSI is enabled, via a console-ops VTABLE seam + an ENABLE gate (the kernel binds the vtable + `sysinit_ansi_enabled`; int21.c stays decoupled from console.c/sysinit.c). kmain binds the adapters -- CRITICALLY `put_char` fans to console AND serial like `int21_con_sink`, so serial markers survive when ANSI.SYS is loaded (the normal boot loads it). Cursor authority: the console is the single source of truth (the FSM's cursor is untrusted for positioning; save/restore reads the live console cursor). DSR (ESC[6n) deferred (needs a kbd-inject seam). Host gates: `test-ansi-wire` (15 checks, gate mutant) + `test-console` extended 71->91.
- **The FAT-cache share (option B) -- frees the kernel-window headroom.** See Frictions: the committee ratified raising PROGRAM_BASE, but it broke SAMIR, so the orchestrator pivoted to the committee's own better option B. The loader no longer keeps its own 6 KiB whole-FAT buffer (`g_load_fat[12*512]`); it ALIASES the int21 file backend's FAT (`fileio_fat_fat_buffer` -- the SAME mounted volume's FAT, slurped once at `fileio_fat_bind` which kmain calls first). Safe because the loader reads the FAT only during `load_program_from_fat` (the load phase), never while a program runs. `loader_bind_fat_volume` now takes the shared `(fat, fat_len)`; only kmain calls it (host tests use `loader_prepare`, not the FAT path). Frees ~6 KiB kernel .bss -> the kernel fits at the ORIGINAL PROGRAM_BASE 0x30000 with p96i (`_kernel_end` 0x30fc0 -> 0x2f7a0 < 0x2ff00), preserving SAMIR's disjoint heap arena.
- **Boot geometry:** KERNEL_SECTORS 208->224 (Makefile + stage2.asm equate) -- p96i grew the on-disk kernel binary ~5 KiB (ansi.c folded into int21.o + the console extensions); IMG_SECTORS=256 accommodates (Rule 5 -> re-verify Bochs).

## Frictions (the load-bearing story)

1. **The committee ratified a fix that broke a working feature; orchestrator grading (Law 4) caught it.** The shell kernel `_kernel_end=0x30fc0` overran PROGRAM_BASE (0x30000) by ~4.3 KiB after the p96i wiring. An ADR-by-committee (memory-architect / period-purist / north-star-risk lenses) RULED to raise PROGRAM_BASE 0x30000->0x38000 (the sanctioned mechanism, precedent 5pe; a growing resident kernel pushing the TPA up is the authentic DOS model). A subagent executed the raise thoroughly (the full blast radius: spec/memory_map.h, 14 .asm fixture orgs, the SECONDARY scratch pages 0x31000/0x32000 in absdisk/vect/datetime, samir.ld/crt0, test_hardware_spec, hardware.json; made PROGRAM_IMAGE computed). Host gates the committee named were green. **BUT the orchestrator re-graded SAMIR (the real AH=48h disjoint-arena consumer) and it BROKE** -- `test-samir-boot` RED, the SAMIR REPL never started. Root cause: raising PROGRAM_IMAGE shrank the disjoint heap arena ([PROGRAM_IMAGE + image + PROGRAM_BSS_RESERVE(64 KiB), ENV_BLOCK=0x5F000)) from ~48 KiB to ~16 KiB -- below what SAMIR needs. The committee verified images LOAD (PROGRAM_IMAGE_MAX ~155 KiB > SAMIR) but MISSED the heap-arena consequence. **LESSON: the committee decides, but the orchestrator must grade the decision's IMPLEMENTATION against the real consumers, not just the named gates -- the green host gates hid a broken emu consumer (the WL-0038 HARD LESSON, applied to a committee ruling).**
2. **The pivot: revert the raise, do the committee's own option B.** Any PROGRAM_BASE raise that buys lasting kernel headroom squeezes SAMIR's heap (the tension is fundamental). Option B (share the dual 6 KiB FAT caches -- which all three lenses called an inauthentic redundancy worth removing) frees 6 KiB of KERNEL .bss instead, so the kernel fits at the ORIGINAL base, preserving SAMIR's arena entirely. The raise was reverted (git checkout of the ~30 raise-files, keeping p96i's), and B implemented via the EXISTING `fileio_fat_fat_buffer` accessor (the SAMIR/sysinit precedent for reusing the FAT buffer to keep .bss clear of PROGRAM_BASE). No host-test coupling (only kmain binds the FAT path).
3. **A host-stub signature drift.** loader.c has TWO `loader_bind_fat_volume` -- the kernel one and a HOST stub (the FAT load is kernel-only). The new shared-cache signature had to update BOTH; the host aggregate caught the stub conflict (compile RED) -> fixed.
