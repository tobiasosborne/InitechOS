## What changed

**Lane 1 -- the render path (kmain.c, behind `-DBOOT_FLAIR_SHELL`):** after the
FLAIR-HEAP-OK gate: `flair_heap_init` over the 4 MiB extended window -> allocate
EVERYTHING from the FLAIR heap (scene, window stores, menus, dialog, every
region's rows[]/x_pool, the 640x480 indexed-8 offscreen) -- NOT kernel BSS (the
~17 KiB runway, bead 5dr8) -> build the scene IDENTICAL to test_shell.c ->
`shell_render` into the 8bpp offscreen -> PRESENT to the live LFB (8bpp: DAC the
FLAIR CLUT + index-copy; 24/32bpp: `flair_palette_rgb` per pixel) -> emit
`FLAIR-DESKTOP` -> halt. A shared `flair_palette_rgb` inline added to
spec/assets/palette.h (emitted by the palette generator, reproducible), a
line-for-line peer of the harness `render_palette_rgb` so the live desktop and
the oracle agree by construction (Law 2). New `KERNEL_FLAIRSHELL_*` kernel +
build/flair_desktop.img. `_kernel_end=0x34bc0` (45 KiB under the 0x40000 ASSERT).

**Lane 2 -- the screendump oracle (tools/ppm_flair_check.c + Makefile):** boots
flair_desktop on QEMU, screendumps after FLAIR-DESKTOP, asserts the desktop
STRUCTURE (seafoam + TWO stacked menu bars [0,20)/[20,40) + window pinstripe
chrome + the modal FILE COPY band occluding the windows) at tol +/-2 against the
exact flair_palette_rgb values. `test-flair-desktop` + `test-flair-desktop-mutant`
(3 SHELL_MUTATE_* legs each RED, Rule 6) into TEST_EMU_GATES; a separate
`test-flair-desktop-bochs` serial leg.

## The display-mode fork + the committee (no gridlock)

Grading surfaced that the desktop renders only on QEMU's 640x480x32 VBE: Bochs
ENOMODEs that and falls to 320x200x8 mode-0x13, where the kmain 640x480 guard
fires (fail-loud) -- so the 8bpp DAC present path is correct but UNREACHED
in-tree. This intersects the operator's same-day 486+/period-SVGA note (fgs1).
The committee ruled (re30.3-chair):
- **(A)** dual-depth strategy of record: request priority 32->24->8(VESA 0x101,
  period-authentic)->mode-0x13 fail-loud floor; OD-2 indexed-8 offscreen
  unchanged; AM-6 host-model-per-mode (NOT a QEMU pin).
- **(B)** KEEP the 8bpp present path (correct-but-unreached, not a Rule-3
  bandaid); add a "do NOT delete" pointer comment; do NOT touch stage2 this
  commit (gold-plating the riskiest boot path for zero M3.1 gain).
- **(C)** M3.1 closes GREEN on the QEMU truecolor leg + Bochs boot-accuracy
  serial; 8bpp/0x101 is an M4 obligation, tracked.

## Frictions / lessons

- A subagent's dev screendump was wiped by a later `make clean`; the orchestrator
  regenerated + Law-4 eyeballed it (build/flair_grade.png) -- the milestone's
  decisive grade is visual, and it IS the frame.
- The host 8bpp coverage (render.c) is a SEPARATE reimplementation; the live-LFB
  DAC-and-row-copy wrapper has ZERO coverage today -- a real Rule-6 gap that
  initech-2gva must close with a REACHING test, not a branch-adjacent one.
- Storage discipline mattered: putting the scene/regions in kernel BSS would
  have busted the 0x40000 ASSERT; all-heap kept _kernel_end at 0x34bc0.
