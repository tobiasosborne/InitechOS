**Codename:** STAPLER  ·  **Version:** 0.3 (Draft)  ·  **Owner:** Tobias  ·  **Date:** 2026-06-08  ·  **Status:** Stack simplified to a C-only factory; formats locked

> *"The screenshot is the spec."*

## 0. TL;DR

A bootable, period-plausible operating system for **emulated 386+ PCs**: a **DOS 3.3 personality** fused with a **System 7 Toolbox** — the exact chimera visible in the *Office Space* "Saving tables to disk…" frame (Macintosh window chrome, DOS drive letters, a Windows hourglass). It really boots. The windows really work. It ships a **dBASE-compatible database that really runs**.

**North star:** a Borland-style single-pass **Pascal compiler** (`Turbo Initech`) with an integrated Turbo-Vision IDE that is **self-hosting** and can **recompile itself — and the Pascal programs it compiles — from inside the OS**. (The OS itself is C, per ADR-0002, and is rebuilt by the factory cross-toolchain.)

It is built by an **agent swarm whose fitness function is the emulator itself**: debug and error signals come from QEMU/Bochs/86Box; success is measured by boot milestones, pixel-fidelity against the film stills, and differential conformance against real period software.

**The whole point (acceptance test):** a person who used early-90s Mac and DOS software boots InitechOS and *cannot immediately tell it isn't a real period product* — down to the **Initech-teal (`#8DDCDC`) desktop and the System-7 decomp-golden chrome palette** (WL-0053; supersedes the earlier seafoam render), the Chicago and Geneva bitmaps, the folder icons, and the pie chart whose slices sum to 116%.

## 1. Vision & North Star

Three commitments, in priority order.

**1.1 Fidelity (the Turing test).** The product is "done" when an informed period user is genuinely unsure whether InitechOS is the software used on set. The reference frame (Appendix A) is reproduced not as a bitmap but as a *live, interactive window arrangement* generated by a real Toolbox. Note the delicious inversion: the original film screens were Photoshop mock-ups (the giveaway is the top menu bar — `File Edit Image Layer Select View Window Help` is Photoshop's, not an OS's). InitechOS *out-reals the prop* by genuinely having those apps with those menus.

**1.2 Self-hosting compiler.** A Borland-like Pascal compiler + blue IDE (`Turbo Initech`) that compiles itself — and the Pascal programs it compiles — from inside the OS, reaching a bit-identical reproduction fixed point (§7). The OS itself is C (ADR-0002) and is rebuilt by the factory cross-toolchain; the self-host fixed point concerns Turbo Initech, not the C kernel/Toolbox. The finale demo: boot the machine, open the IDE, load the Turbo Initech (Pascal) source, hit Compile, and get a new self-reproduced compiler — the certificate that the resident compiler is real.

**1.3 Anti-vibeOS thesis.** The inspiration (Sanderson's *vibeOS*, where every UI is hallucinated by an LLM and nothing is backed by code) is the *opposite pole*. InitechOS is deterministic and real all the way down; that is what earns the joke. A hidden, opt-in `VibeMode` (§13) is the only nod to the other pole, and it exists precisely to make the contrast legible.

## 2. Goals / Non-Goals

**Goals**

- Boots on emulated 386/486 from a disk image; no host OS underneath.
- DOS 3.3 personality: FAT12/16, `INT 21h`-style API, program loader, command shell.
- System-7-style Toolbox: region-based graphics, windows, menus, controls, events, dialogs, resources.
- A dBASE III+/IV-compatible database app (`InitechBase`) with real `.dbf`/index I/O and a working xBase interpreter.
- A self-hosting single-pass Pascal compiler (`Turbo Initech`) with a resident Turbo-Vision IDE.
- Asset-level authenticity: Chicago/Geneva bitmaps, icons, palette, chrome metrics — extracted from the film.
- An oracle-driven build pipeline suitable for an agent swarm.

**Non-Goals (explicit)**

- 8088/80286 real-mode purism. *Dropped by decree — no fond memories of segment:offset awkwardness.* 386+ only.
- Running on bare metal modern hardware (emulated targets only; real-iron boot is a stretch, not a requirement).
- SMP, preemptive multitasking, paging-as-VM, POSIX, protected inter-process isolation.
- In-kernel TCP/IP or TLS (the optional LLM sidecar in §13 uses a host proxy over serial).
- Performance. These are slow machines; codegen quality is irrelevant.

## 3. The Fidelity Bar — Definition of Done

The reference frame (and the short on-screen sequence around it) is a *fixture*, not inspiration. Its forensic tells become acceptance criteria (full table in Appendix A): the Mac chrome, the DOS "Drive A," the hourglass-not-wristwatch cursor, the Photoshop menu bar on the paint app, the `570-` trailing-minus negative format, and the pie chart summing to 116%. The bar for the desktop milestone is simply: **it looks and feels like the frame and the short clip** — a person who knows the era says "yes, that's it." SSIM against the film still is a *useful guide* the harness reports per-window to drive the agents toward fidelity, **not a hard numeric gate**; eyeballing the live, draggable arrangement with menus working is the real judge.

Assets lean on the frames where they can (§10) — palettes, glyph bitmaps, icon sprites, chrome metrics measured from stills — but the still is low-res and compressed, so in practice most glyph strikes are **hand-authored and back-checked against the frame** rather than pixel-extracted. "Screenshot is the spec" is the spirit, not a literal pixel-recovery promise.

## 4. System Architecture

A graphical Toolbox layered on a DOS-like base. This is historically coherent — it is what GEM and Windows 1–3 were (a Mac-flavoured GUI over DOS) — and the reference frame *is* this chimera.

```
┌──────────────────────────────────────────────────────────────┐
│  APPS                                                          │
│   InitechCalc · File Manager · InitechPaint · InitechBase      │
│   Turbo Initech (compiler + Turbo-Vision IDE)                  │
├──────────────────────────────────────────────────────────────┤
│  INITECH TOOLBOX  (the "Mac" chimera, codename FLAIR)          │
│   Window · Menu · Control · Event · Dialog · Resource managers │
│   ATKINSON region engine  │  bitmap fonts (Chicago/Geneva)     │
├──────────────────────────────────────────────────────────────┤
│  INITECHDOS  (the "DOS 3.3" personality, codename MILTON)      │
│   INT-21h-style API · FAT12/16 · program loader · console      │
├──────────────────────────────────────────────────────────────┤
│  PLATFORM                                                      │
│   boot (real → 32-bit protected/flat) · VESA LFB · PS/2 · PIT  │
├──────────────────────────────────────────────────────────────┤
│  EMULATED 386+  (QEMU · Bochs · 86Box)                         │
└──────────────────────────────────────────────────────────────┘
```

**Two compilers, never conflated.** The OS itself (kernel, InitechDOS, Toolbox, bundled apps for the current release) is **C** (ADR-0002), built and rebuilt by the **factory cross-toolchain** (target `i686-elf`; interim host gcc per CDR-0001). The *seed cross-compiler* (C, on the dev machine, Pascal→x86) is the **genesis of Turbo Initech**, not the OS bootstrap: it exists to reach the basin of attraction of the resident compiler's fixed point (§7). The *resident compiler* (`Turbo Initech`) is **Pascal**, runs on InitechOS, and is the self-hosting in-universe artifact. "Recompile from inside the OS" means **Turbo Initech and the Pascal programs it compiles** — not the C kernel/Toolbox.

## 6. Subsystem Specifications

Each subsystem names its **spec** and its **oracle**. No subsystem ships without a mechanical oracle; that discipline is the project.

### 6.1 InitechDOS kernel (codename MILTON)

Responsibilities: boot handoff, protected-mode bring-up, LFB init, PS/2 + PIT IRQs, FAT12/16 read (write optional), program loader, an `INT 21h`-flavoured syscall surface (open/read/write/seek/close, alloc, console I/O, dir enumeration), and a `COMMAND.COM`-alike shell (`DIR`, `TYPE`, `CD`, run a program from `A:`).
Oracle: serial boot-stage markers; QEMU `-d int,guest_errors,cpu_reset`; FAT operations diffed against a host `mtools`/Python reference on identical disk images.

### 6.2 ATKINSON region engine — *the load-bearing math*

QuickDraw-style regions are the spine of the GUI; clipping, window overlap, `visRgn`/`clipRgn`, and update regions all reduce to region algebra.

Model a region as a pixel set `R ⊆ ℤ²`, represented per scanline by a sorted list of **inversion points** `x₀ < x₁ < …` at which membership toggles (a normalized RLE). The set of regions over a bounding rectangle forms a **Boolean algebra** under `∪, ∩, ∖, △` and complement; all ops are implemented by a **scanline merge** of inversion lists. The normal form is the minimal inversion-point set (no redundant/adjacent toggles).

**Correctness oracle (homomorphism property):** for each op `⊕`,
`rasterize(A ⊕ B) = rasterize(A) ⊕_set rasterize(B)`
i.e. rasterization is a Boolean-algebra homomorphism from the region representation to the powerset of pixels over the bounding box. **Property-test in C** with a small randomized generator (random regions over a bounded grid, thousands of cases per op, shrinking on failure): rasterize both sides to bitmaps and assert equality, plus normal-form idempotence (`normalize(normalize(R)) == normalize(R)`) and the algebra identities (De Morgan, associativity, `A ∖ A = ∅`). This brute-force differential against the pixel ground truth is the spine; it needs no proof assistant to be decisive, and it pays for itself across the entire Toolbox.

### 6.3 InitechToolbox (codename FLAIR)

Window Manager (drag, z-order, update regions via §6.2), Menu Manager (pull-down menus, the menu bar — including the Photoshop-exact bar for InitechPaint), Control Manager (buttons, scrollbars, the progress bar), Event Manager (a cooperative `WaitNextEvent`-style loop), Dialog Manager (modal dialogs — the FILE COPY box lives here). Chrome metrics (pinstripe title bars, close/zoom boxes, scrollbar thumbs) come from extracted fixtures.
Oracle: chrome-only renders scored by SSIM against fixture crops; region property suite green; event-trace replays.

### 6.4 Resources & Assets

A resource fork analogue (typed, ID'd blobs: `FONT`, `ICON`, `PALT`, `WIND`, `MENU`). Chicago (system/dialog font — "Saving tables to disk…" is Chicago) and Geneva 9 (cell font), as **clean-room bitmap strikes hand-authored to match the on-screen glyphs** and back-checked against the frame (the still is too low-res for pure pixel extraction). Free clones (ChiKareGo, FindersKeepers, ChicagoFLF) are reference only; ship clean-room bitmaps (see IP note, §12).

### 6.5 Desktop & Apps

- **InitechCalc** — editable grid, Geneva 9, the Initech ledger from the frame; a pie chart whose slices sum to 116% (enforced by test, Appendix B); `570-` trailing-minus cell format.
- **File Manager** — the "Drive A Files" window with folder icons.
- **InitechPaint** — exists primarily to carry the Photoshop-exact menu bar so the frame is self-consistent.
- **FILE COPY dialog** — the comedic centerpiece; progress bar fills at the rate set in `michael_bolton.conf` (Appendix B).

Oracle: M5 reproduces the *entire* reference frame as a live, interactive arrangement that looks and feels like the film clip — per-window SSIM reported as a fidelity guide, not a hard cutoff.

### 6.6 InitechBase — the dBASE-alike that really works (codename SAMIR)

Runs as a **character-mode xBase program in an 80×25 text-console window inside the Mac desktop** — the DOS-box-in-the-GUI metaphor made concrete, with authentic CP437 / Turbo-Vision rendering.

Scope: `.dbf` reader/writer (field types C/N/D/L/M, the documented header), `.dbt` memos, `.ndx`/`.mdx` B+-tree indexes, a tree-walking interpreter for the dot-prompt subset (`USE`, `APPEND`, `LIST`, `LOCATE`/`SEEK`, `SET INDEX`/`SET ORDER`, `DO WHILE…ENDDO`, `IF…ENDIF`, `REPLACE`), and `@ row,col SAY … GET … / READ` full-screen forms.
Oracle: real dBASE III+/IV under DOSBox-X / 86Box — **whatever it takes to make dBASE actually work.** The bar is **interoperable round-trips** (a `.dbf`/`.mdx` we write is read back correctly by real dBASE, and vice versa) and **differential program output** (generate xBase programs, run both, diff). Round-trips are compared on the *meaningful* bytes — header fields like the last-update date and reserved/language-driver bytes are normalized before diffing, since exact reproduction of those is neither possible nor the point. The xBase type-coercion table (string/number/date/logical, `SET EXACT`) is small; capture it as a data table and fuzz the evaluator against the real interpreter.

> **M6 scope reconciliation (operator decision 2026-06-17; governed by ADR-0008, supersedes the "dBASE IV" / `.mdx` wording above and at §11-M6).** M6 targets **dBASE III PLUS 1.1 ONLY**. The entire ground-truth base (the `../dbase3-decomp` corpus, the golden fixtures, and the live dosbox-x mint harness) is III+ 1.1, and no dBASE IV golden exists. Consequently `.mdx`, the `F` (float) type, the `==` operator, and the `0x8B` IV memo dialect are **deferred to a post-M6 dBASE IV epic** — never half-built or guessed (Law 1). M6 ships `.ndx` only. The M6 Definition of Done reads "differential vs real dBASE **III PLUS 1.1** green; interoperable III+ round-trips." See `docs/adr/ADR-0008-SAMIR-InitechBase-Architecture.md` and `docs/plans/SAMIR-implementation-plan.md`.

### 6.7 Turbo Initech — compiler + IDE (codename TPS)

The OS itself is **C** (ADR-0002); Turbo Initech is the in-universe **Pascal** self-hosting compiler — the North Star app (ADR-0007, pending). Language **Pascal** (Turbo-Pascal-flavoured subset with `pointer`, `record`, inline `asm`, units). Rationale: it is the authentic Borland north star, Wirth designed Pascal for single-pass compilation, and Pascal-with-inline-asm is period-honest as a systems language. Single-pass front end; **stack-machine codegen** (expression evaluation to x86, fixed register conventions, *no optimizer, no register allocator*) — small and sufficient. Emits freestanding x86 executables: it compiles **itself** and the **Pascal user programs** authored on InitechOS. The IDE is the Turbo-Vision blue screen, running as a resident InitechOS app (editor + compile + run), file I/O via InitechDOS.
Oracle: differential vs Free Pascal on the shared subset (compile a corpus, run outputs, diff); and the **self-host fixed point** (§7), which is a brutal, self-correcting integration test — miscompile yourself and the rebuild won't converge or won't boot. The fixed point concerns Turbo Initech (and the Pascal programs it compiles), **not** the C kernel/Toolbox, which is rebuilt by the factory cross-toolchain.

## 7. The Bootstrap — Self-Hosting as a Fixed Point

**Scope.** This section concerns **Turbo Initech** (the resident Pascal compiler) and the Pascal programs it compiles — *not* the C kernel/Toolbox, which is built and rebuilt by the factory cross-toolchain (ADR-0002; interim per CDR-0001). The Φ fixed point and DDC below are the certificate for Turbo Initech's self-reproduction, not for the C OS.

Let `S` be the set of compiler binaries runnable on InitechOS, and `src` the canonical compiler source (in Pascal). Define the compilation operator

```
Φ : S → S ,   Φ(K) = K(src)        ("run binary K on the canonical source → new binary")
```

Self-hosting correctness is: **`Φ` has a fixed point `K*` reachable from the seed, with `Φ(K*) = K*` bit-for-bit** (idempotence under reproducible build).

Bootstrap is the iteration. With seed cross-compiler `X` (runs on host, targets InitechOS):

```
K₁ = X(src)          # seed builds the resident compiler (cross)
K₂ = K₁(src)         # resident compiler rebuilds itself (now hosted on InitechOS)
K₃ = K₂(src)         # …
require  K_{n+1} == K_n   (bit-identical)
```

In practice the **two-stage** result is the certificate: `K₁` (emitted by a *different* compiler, the seed) may differ byte-wise from `K₂`, but `K₂ == K₃` is the self-reproduction proof. Reproducible builds (no timestamps, deterministic symbol ordering) make the diff meaningful.

**Trusting-trust caveat.** A fixed point can harbour a self-perpetuating backdoor (Thompson, *Reflections on Trusting Trust*). The antidote in scope is **diverse double-compilation** (Wheeler's DDC — compile `src` with an independent seed and check the resulting `K*` matches). DDC is cheap and decisive here; formal semantic-preservation proofs are explicitly out of scope.

## 8. Oracle & Test Infrastructure — debug signals from the emulator

| Emulator | Role | Signals consumed |
|---|---|---|
| **QEMU** (i386) | fast dev loop | gdb stub (`-s -S`) + symbols; QMP `screendump`; `-serial` printf; `-d int,guest_errors,cpu_reset`; triple-fault detect |
| **Bochs** | accuracy + debug | built-in debugger; strict real→protected transition checking |
| **86Box / PCem** | period authenticity + reference | cycle-accurate 486 + period VGA BIOS (Cirrus/ET4000); runs **real DOS 3.3 + real dBASE** to mint golden files |

Running all three is **differential emulation** — it catches QEMU-isms, which matters because "could have run on real PCs of the era" is a hard requirement, not a vibe.

**Gate vector (the dashboard the agents climb).** Not a single magic scalar to optimize — the milestones gate each other, so this is a per-subsystem set of pass/fail gates plus continuous guides:

```
boot_stage_reached            # serial milestones — discrete gates per M1/M2
SSIM(screendump, fixture)     # continuous *guide* toward chrome/desktop fidelity (not a hard cutoff; harness/ssim.c PLANNED -- NOT YET BUILT, ADR-0010/HER-11)
differential_pass_rate        # InitechBase vs real dBASE; Turbo Initech vs fpc — gate at 100% on the shared corpus
self-host fixpoint converges  # boolean gate — bit-identical K₂ == K₃
panics / guest_errors         # must be zero on the green path
```

Each subsystem advances on its own gate; SSIM is reported per-window to point agents in the right direction, not summed into a reward. Golden-file generation, region property suites, differential corpora, and the fixpoint check all run locally on every change.

## 9. Agent-Swarm Methodology

The non-obvious claim: this is *more* swarm-tractable than a game clone, because every subsystem has a mechanical oracle and therefore a gradient. Discipline:

- **Lock specs as data.** Region algebra, xBase coercion table, chrome metrics, asset sheet, hardware contract — all live as versioned data (plain JSON / C headers, kept dead simple) that the swarm consumes. The locked spec is the contract; mush is what you get without it.
- **Gate-driven, not reward-driven.** Drive the work off the §8 gate vector — each subsystem advances when its oracle goes green. No global scalar to game.
- **One oracle per subsystem.** No "looks right." Either it diffs against a reference or it isn't done.
- **Meta-work is the project.** Building the oracle harnesses, the golden files, and the extracted asset sheet *is* the work. Skimp there and the swarm sprawls — particularly on the Toolbox, which without the region-algebra anchor and the chrome-SSIM gate degenerates into thousands of lines of plausible nonsense.

## 10. Asset-Extraction Pipeline — "screenshot is the spec," literally

Frame stills → (CV-assisted, human-verified) → palette quantization, chrome metric measurement (title-bar height, pinstripe period, scrollbar widths), icon sprite extraction, and glyph recovery *where the resolution allows*. Realistically the still is low-res and compressed: palette and chrome metrics extract cleanly; **most glyph strikes are hand-authored to match the on-screen glyphs and back-checked against the frame**, not recovered pixel-perfect. Output: a verified sprite/metric/palette sheet checked into the repo as the source of truth for §6.4 and the SSIM-guide fixtures in §3. Tooling is small C utilities + plain-data sheets; a clean, parallelizable workstream.

## 14. Tooling & Stack

The 2026 agent factory produces a 1991 reproducible artifact; the stack lives in the factory, the artifact stays period-authentic. **Decree: the factory is C, and only C** — one language to bootstrap, no proof assistants, no JVM, no extra runtimes. Simplicity here is a feature, because polyglot tooling is itself a spec-drift vector.

- **Build orchestration + oracle harness:** C (+ thin shell/`make` glue) — drives QEMU/Bochs/86Box, computes SSIM, runs the differential suites, dispatches agent work. No bandit/population loop; gate-driven (§8).
- **OS implementation language:** **C** (per ADR-0002) — the bootloader, the InitechDOS kernel (`IO.SYS`/`INITDOS.SYS`/`COMMAND.COM`), the Toolbox (FLAIR), and the bundled apps for the current release. Target toolchain is **`i686-elf` cross-compilation** (ADR-0002); the **current interim toolchain** is host `gcc -m32 -ffreestanding -nostdlib` + `nasm` + `ld`, an accepted time-limited deviation recorded in **CDR-0001** (to be remediated when development moves to a more capable device). The factory remains C-only.
- **Seed cross-compiler:** C. A from-scratch single-pass Pascal compiler, hosted on the dev machine, targeting freestanding InitechOS/x86. - **Artifact languages:** the OS (kernel, InitechDOS, Toolbox, bundled apps for the current release) is **C** (ADR-0002). **Pascal** is the language of **Turbo Initech** — the resident self-hosting compiler (ADR-0007, pending) — and of the user programs it compiles; it is the in-universe Borland north star (§6.7).
